# Instructions and uncertainties

Not everything extracted from a conversation turn is a fact about the world. Some turns tell the agent how to behave; others contain statements that cannot be confidently interpreted. Spectron handles these as two specialised record types: **instructions** and **uncertainties**.

## Instructions

An **instruction** is a behavioural directive extracted from a turn – something the user has told the agent to do or not do, persistently, for future interactions.

Common examples:

- "Always respond in bullet points."
- "Never use technical jargon – I'm not an engineer."
- "From now on, track when I'm being sarcastic and label it."
- "Only respond in French."
- "Keep responses under three sentences."

Instructions are distinct from ordinary preferences or attributes. They are active behavioural constraints that the agent is expected to enforce on every subsequent response within the relevant scope.

### Schema

```surql
DEFINE TABLE instruction SCHEMAFULL;
DEFINE FIELD label ON instruction TYPE string;
DEFINE FIELD description ON instruction TYPE string;
DEFINE FIELD active ON instruction TYPE bool DEFAULT true;
DEFINE FIELD scope ON instruction TYPE array<record<scope_attribute>>;
DEFINE FIELD source_turn ON instruction TYPE option<record<turn>>;
```

- `label` – a short name for the instruction, generated by the extraction pipeline (e.g. "Bullet-point responses")
- `description` – the full directive as understood by the pipeline
- `active` – whether the instruction is currently in effect; defaults to `true`
- `scope` – the scope dimensions this instruction applies to
- `source_turn` – the turn that produced the instruction, for traceability

### Extracting instructions from turns

The extraction pipeline identifies instructions from imperative language, explicit scope markers ("always", "never", "from now on"), and contextual signals. When an instruction is detected, it is added to the extraction document alongside any entities and attributes:

```json
{
  "instructions": [
    {
      "label": "Bullet-point responses",
      "description": "Always respond using bullet points, never flowing prose"
    },
    {
      "label": "Track sarcasm",
      "description": "Track personal details and include a visual label when the user is being sarcastic"
    }
  ]
}
```

### The active toggle

Instructions can be deactivated without being deleted. Deactivation is appropriate when a user revokes a previous directive ("actually, you can stop using bullet points") or when an instruction is no longer relevant.

```python
await memory.instructions.update(
    instruction_id="instruction:bullet_points",
    active=False,
)
```

```typescript
await memory.instructions.update({
    instructionId: "instruction:bullet_points",
    active: false,
});
```

When a turn contains a revocation, the extraction pipeline sets `active = false` on the matching instruction automatically. The record is preserved – you can inspect the history of directives a user has given over time.

### Querying active instructions

Agents should retrieve active instructions before generating each response and incorporate them into their system prompt or response constraints:

```python
instructions = await memory.instructions.list(
    scope={"org": "acme", "user": "christian"},
    active=True,
)

system_prompt = build_system_prompt(base_prompt, instructions)
```

```typescript
const instructions = await memory.instructions.list({
    scope: { org: "acme", user: "christian" },
    active: true,
});

const systemPrompt = buildSystemPrompt(basePrompt, instructions);
```

Active instructions are also included in the `/state` response, so a single call can populate both the entity context and the behavioural constraints for a given interaction.

```python
state = await memory.state(scope={"org": "acme", "user": "christian"})

active_instructions = state["instructions"]
entities = state["entities"]
```

### Instructions in SurrealQL

To query all active instructions for a given scope directly:

```surql
SELECT * FROM instruction
WHERE active = true
AND scope CONTAINSALL $scope_dimensions
ORDER BY created_at DESC;
```

## Uncertainties

An **uncertainty** is a record created when the extraction pipeline encounters something it cannot confidently interpret. Rather than guessing, the pipeline records what it noticed and why it could not resolve it.

Common triggers:

- A statement followed by a contradictory correction ("I'm CEO – well, co-CEO, it's complicated")
- Ambiguous pronoun resolution when multiple entities are in scope
- A claim that directly contradicts a previously recorded fact with no clear indication of which is correct
- Temporal references that cannot be resolved to a specific date
- Sarcasm or humour that may or may not contain genuine information

### Schema

```surql
DEFINE TABLE uncertainty SCHEMAFULL;
DEFINE FIELD about ON uncertainty TYPE string;
DEFINE FIELD reason ON uncertainty TYPE string;
DEFINE FIELD resolved ON uncertainty TYPE bool DEFAULT false;
DEFINE FIELD source_turn ON uncertainty TYPE record<turn>;
```

- `about` – a description of what the uncertainty concerns (e.g. "job title", "current city")
- `reason` – why the pipeline could not resolve it (e.g. "User made a joke then appeared to correct themselves")
- `resolved` – whether a subsequent turn has clarified the ambiguity
- `source_turn` – the turn that triggered the uncertainty

### How uncertainties are resolved

When a later turn provides information that resolves an open uncertainty, the extraction pipeline sets `resolved = true` on the matching record:

```surql
UPDATE uncertainty SET resolved = true
WHERE about = "job title"
AND resolved = false;
```

Resolution is automatic when the pipeline confidently extracts a value for the same topic in a subsequent turn. Manual resolution is also available via the API for cases where the pipeline does not detect the connection:

```python
await memory.uncertainties.resolve(uncertainty_id="uncertainty:abc123")
```

### Uncertainties in the state API

Open uncertainties are surfaced in the `/state` response under the `unknowns` key:

```json
{
  "unknowns": [
    {
      "id": "uncertainty:abc123",
      "about": "job title",
      "reason": "User made a joke about being CEO, then appeared to correct themselves",
      "resolved": false,
      "source_turn": "turn:xyz789"
    }
  ]
}
```

Agents can use this information to ask clarifying questions when appropriate:

```python
state = await memory.state(scope={"org": "acme", "user": "christian"})

if state["unknowns"]:
    for unknown in state["unknowns"]:
        # Optionally prompt the user to clarify
        print(f"Unclear: {unknown['about']} – {unknown['reason']}")
```

### The relationship between uncertainties and attributes

When an uncertainty is detected alongside a partial extraction, Spectron may still write the extracted attribute – but it will be linked to the uncertainty record via the `source_turn`. This means the attribute is present in the store but flagged as potentially unreliable.

Current-state queries return these attributes normally. If your application requires higher confidence, you can filter for attributes whose source turn has no unresolved uncertainties:

```surql
SELECT a.* FROM attribute AS a
WHERE a.entity = $entity_id
AND a.superseded_by IS NONE
AND (
    a.source_turn IS NONE
    OR NOT (
        SELECT id FROM uncertainty
        WHERE source_turn = a.source_turn
        AND resolved = false
    )
);
```
