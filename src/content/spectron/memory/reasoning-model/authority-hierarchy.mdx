# Authority when pillars meet

Spectron’s memory model is built around **eight pillars** of agent memory – not a flat “two layers” split. Two of those pillars matter most for day-to-day API behaviour:

- **[Authoritative](/docs/spectron/architecture/eight-pillars-and-categories)** – curated artefacts (`source.kind = "document"`, operator `upsert`): manuals, policies, product data, structured exports. Higher default **trust**.
- **[Experiential](/docs/spectron/architecture/eight-pillars-and-categories)** – conversational input and everything derived from it (`turn`, `reflect`, `elaboration`, `consolidation`, …). The **Experiential** pillar includes the **six memory categories** (episodic turns plus identity, knowledge, context, instructions, uncertainty) described in [Memory categories](/docs/spectron/mental-model/memory-categories).

**Reconciliation** (its own pillar) is the single function that merges streams: supersession, **`uncertainty`** rows, and trust/calibration logic are identical whether facts arrived from a document or a turn.

Physically, both streams are rows in the **same** SurrealDB substrate – see [Unified substrate and authority](/docs/spectron/mental-model/two-layer-architecture).

## Curated truth is never silently overwritten

**Authoritative** content does not change because a user said something different. When an experiential assertion conflicts with curated knowledge, Spectron:

1. Keeps the curated row unchanged.
2. Stores the user’s belief in the experiential graph with provenance intact.
3. Surfaces the clash (for example via **`uncertainty`** and/or conflict flags on the attribute).

The user’s assertion remains valuable conversational context – it simply does **not** become canonical truth.

This prevents **hallucination amplification**: incorrect chatter never silently rewrites product, policy, or compliance data.

## Worked example

Your authoritative knowledge base contains:

```json
{
  "id": "knowledge:[\"Policy\", \"returns\"]",
  "layer": 0,
  "content": {
    "duration_days": 30,
    "condition": "unused and in original packaging"
  }
}
```

> The `layer` field in JSON examples is illustrative of API grouping; the conceptual model is the **Authoritative** pillar, not a separate database.

A user tells the agent: "Your return policy is 60 days."

The extraction pipeline detects a conflict and writes to **experiential** memory:

```json
{
  "id": "attribute:user_belief_returns",
  "entity": "entity:policy_returns",
  "key": "duration_days",
  "value": "60",
  "layer": 1,
  "contradicts_l0": true,
  "scope": { "org": "acme", "user": "alex" },
  "source_turn": "turn:xyz789"
}
```

> `contradicts_l0` is the **legacy API flag name** for “conflicts with authoritative (curated) knowledge.” New integrations should read it as “contradicts authoritative content” regardless of the suffix.

The curated record is unchanged. The `/state` API can surface both values so the agent can acknowledge the user’s belief while stating the policy that actually applies.

## Attribute resolution order

When both curated and experiential values exist for the same key:

1. **Authoritative** values are returned as ground truth for factual questions (price, policy, specification).
2. **Experiential** values are returned alongside for personalisation, with conflict flags when they disagree.

Agents should **trust curated answers for compliance** and use experiential rows for preferences, history, and stated beliefs.

## The `resolves_to` mechanism

Experiential **entities** can link to typed **knowledge** nodes via `resolves_to` so a mention of “AirPods Pro” in chat joins the same real-world object as the catalogue row. See [Cross-layer linking](/docs/spectron/knowledge/retrieval/cross-layer-linking).

Resolution combines LLM suggestions, embedding checks, background scans, and manual overrides – all without breaking the single-substrate model.

## Practical guidance

- **Facts with regulatory weight** – always ground in **Authoritative** content.
- **Personalisation** – lives in the **Experiential** pillar (identity, knowledge, context, instructions).
- **Contradictions** – surface them honestly; never pick silently.
- **Context scope** – curated catalogue updates propagate to every agent in the Context; experiential rows remain scoped per user/team/project as you configure.

For the full pillar list (reconciliation, elaboration, reflection, consolidation, calibration, collective), see [Eight pillars and six categories](/docs/spectron/architecture/eight-pillars-and-categories).
