# Authority hierarchy

Spectron maintains two distinct layers of information with different authority levels. Understanding which layer wins in a conflict — and why — is essential for designing agents that behave predictably when users assert things that contradict what you know to be true.

## Layer 0 always wins

**Layer 0 is the authoritative layer.** It holds knowledge your team has explicitly loaded: product catalogues, policies, documentation, structured reference data. This layer does not change based on what users say.

**Layer 1 is the conversational layer.** It holds what agents and users have told each other. This layer evolves with every interaction.

When a user asserts something that conflicts with a Layer 0 fact, Spectron records the user's belief in Layer 1 but does not overwrite Layer 0. The user's assertion is preserved — it may be relevant conversational context — but it does not become the authoritative truth.

This prevents a class of failure that is common in simpler memory systems: **hallucination amplification**, where an agent hears an incorrect assertion from one user, stores it as fact, and then repeats it to other users or uses it to answer future questions.

## What happens when Layer 1 contradicts Layer 0

Consider a concrete example. Your Layer 0 knowledge base contains:

```json
{
  "id": "knowledge:[\"Policy\", \"returns\"]",
  "layer": 0,
  "content": {
    "duration_days": 30,
    "condition": "unused and in original packaging"
  }
}
```

A user tells the agent: "Your return policy is 60 days."

The extraction pipeline detects a contradiction with Layer 0 and writes the following to Layer 1:

```json
{
  "id": "attribute:user_belief_returns",
  "entity": "entity:policy_returns",
  "key": "duration_days",
  "value": "60",
  "layer": 1,
  "contradicts_l0": true,
  "scope": { "org": "acme", "user": "alex" },
  "source_turn": "turn:xyz789"
}
```

The Layer 0 record is unchanged. The `/state` API surfaces both pieces of information and clearly distinguishes them:

```json
{
  "facts": {
    "layer_0": {
      "Policy.returns.duration_days": 30
    },
    "layer_1": {
      "Policy.returns.duration_days": {
        "value": "60",
        "contradicts_l0": true,
        "note": "User asserted this in turn:xyz789; Layer 0 states 30 days"
      }
    }
  }
}
```

The agent can use this distinction to respond appropriately: acknowledging what the user believes while providing the accurate Layer 0 information.

## Attribute lookups and the authority order

When an agent queries for the value of a specific attribute, Spectron resolves it in the following order:

1. **Layer 0** — if an authoritative value exists for this entity and key, it is returned as the primary value
2. **Layer 1** — if a Layer 1 attribute exists for the same key in the current scope, it is returned alongside the Layer 0 value, with the `contradicts_l0` flag if applicable

This means agents always have access to both the ground truth and the conversational context, and can make deliberate choices about how to use each.

## The `resolves_to` mechanism

Layer 1 entities can be linked to their corresponding Layer 0 knowledge nodes via a `resolves_to` relation. This is how a conversational reference to a product, policy, or any named thing connects to the authoritative record for that thing.

```json
{
  "in": "entity:[\"Product\", \"airpods_pro\"]",
  "out": "knowledge:[\"Product\", \"airpods_pro\"]",
  "relation": "resolves_to"
}
```

Once this link exists, queries for the Layer 1 entity automatically surface Layer 0 facts for it. The agent does not need to know in advance which Layer 0 node corresponds to a user's mention — the resolution happens automatically.

### How resolution works

Resolution proceeds in four steps:

1. **LLM suggestion** — during extraction, the LLM is provided with a summary of accessible Layer 0 node titles and types. It proposes `resolves_to` mappings for any Layer 1 entities it believes correspond to Layer 0 nodes.
2. **Embedding verification** — each proposed link is validated by computing the embedding similarity between the Layer 1 entity name and the Layer 0 node title. Links below the configured threshold (default 0.85) are rejected.
3. **Background scanning** — a periodic background job scans Layer 1 entities that have no `resolves_to` link and attempts resolution against the current Layer 0 state. This catches cases where a matching Layer 0 node did not exist at extraction time but was added later.
4. **Manual override** — links can be set or corrected via the entity API when automated resolution produces an incorrect mapping.

```http
PATCH /contexts/{ctx}/entities/{entity_id}
Content-Type: application/json

{
  "resolves_to": "knowledge:[\"Product\", \"airpods_pro\"]"
}
```

## Practical implications for agent design

**Trust Layer 0 for facts.** When an agent needs to answer a factual question — price, policy, specification — it should query Layer 0. Layer 1 provides context about what the user believes or prefers, not ground truth.

**Use Layer 1 for personalisation.** The user's history, preferences, stated attributes, and prior decisions live in Layer 1. These are not overridden by Layer 0 and are essential for personalised responses.

**Surface contradictions explicitly.** When a user's Layer 1 belief contradicts a Layer 0 fact, the recommended pattern is to acknowledge the discrepancy directly rather than silently defaulting to one or the other:

> "You mentioned our return window is 60 days, but our policy is actually 30 days for unused items in original packaging. I want to make sure you have the accurate information."

**Do not rely on Layer 1 for compliance-sensitive facts.** Anything that carries regulatory, contractual, or financial weight should be sourced from Layer 0. A user's claim about a commitment made by your company does not become binding just because it was extracted into Layer 1.

**Layer 0 is scoped to the Context.** All users within a Context share the same Layer 0 knowledge base. Changes to Layer 0 are immediately reflected for all agents operating within that Context. This makes Layer 0 the right place for product updates, policy changes, and any correction that should affect all users simultaneously.
