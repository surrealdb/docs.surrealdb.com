# Cross-layer linking

When a Layer 1 entity corresponds to something in Layer 0 — a product, a policy, a person — Spectron can link the two via a `resolves_to` reference. The agent then has both conversational context from Layer 1 and authoritative facts from Layer 0 for the same real-world thing.

## What `resolves_to` does

Every Layer 1 entity can carry an optional `resolves_to` field pointing to a Layer 0 knowledge node:

```surql
DEFINE FIELD resolves_to ON entity TYPE option<record<knowledge>>;
```

When the link is present, attribute lookups check Layer 0 first for authoritative values, then Layer 1 for conversational context. If the two conflict, Layer 0 wins.

### Example

A user mentions "I bought AirPods Pro last week." Spectron extracts:

- **Layer 1 entity**: `entity:["Product","airpods_pro"]` with attribute `purchased = "last week"`
- **Layer 0 node**: `knowledge:["product","airpods_pro_2"]` with `price = 249`, `return_policy = "30 days"`, `warranty = "1 year"`

With `resolves_to` set, the agent can answer "Can I still return them?" by combining:
- *User's memory* (L1): purchased last week
- *Authoritative policy* (L0): 30-day return window

Neither source is sufficient alone. Together they give a correct, grounded answer.

## How resolution works

Resolution follows a four-step process:

### 1. Extraction-time suggestion

During extraction, the LLM is provided with a summary of accessible Layer 0 knowledge node titles and kinds within the session's scope. The extraction prompt invites the LLM to suggest `resolves_to` mappings for entities it recognises as matching Layer 0 nodes.

### 2. Embedding verification

Suggestions are verified by embedding similarity. The entity's embedding is compared against the embeddings of candidate knowledge nodes. Only mappings above a configurable similarity threshold (default 0.85 cosine similarity) are accepted. This prevents the LLM from hallucinating links to nodes that don't exist or that are only loosely related.

### 3. Background scan

A background job periodically scans Layer 1 entities that have no `resolves_to` link against Layer 0 nodes. This catches matches for entities that were created before a relevant Layer 0 node existed — for example, a product entity that predates the product catalogue being loaded.

### 4. Manual override

Links can be set or corrected via the entity API:

```http
PUT /v1/{context_id}/entities/{type}/{name}
Content-Type: application/json

{
  "resolves_to": "knowledge:product:airpods_pro_2"
}
```

Python SDK:
```python
entity = await memory.entities.get("Product", "airpods_pro")
# entity.resolves_to is the current link, or None
```

## Querying via the entity API

```python
entity = await memory.entities.get("Product", "airpods_pro")
print(entity.resolves_to)  # "knowledge:product:airpods_pro_2" or None
```

```javascript
const entity = await memory.entities.get("Product", "airpods_pro");
console.log(entity.resolvesTo);
```

The entity response includes the resolved Layer 0 node's summary when `resolves_to` is set.

## How the agent uses both layers

When an agent retrieves context and a Layer 1 entity resolves to a Layer 0 node, Spectron's query resolver:

1. Retrieves the Layer 1 entity's conversational attributes (what the user said, what was learned in sessions).
2. Retrieves the Layer 0 node's authoritative content (price, policy, specifications).
3. Merges them into the formatted context block, distinguishing L0 facts from L1 memories.
4. Flags any contradictions (L1 attribute contradicting L0 value) so the agent can handle them appropriately.

This combined context block is returned by `session.context()` and the `memory_recall` MCP tool.

## Scope and access

`resolves_to` links are only created when the Layer 0 node is accessible from the session's scope. An entity created in scope `{org: "apple"}` can only resolve to Layer 0 nodes also accessible from that scope. Nodes uploaded with empty scope `{}` are globally accessible and can be linked from any entity.

## Auditing links

To find all Layer 1 entities linked to a specific Layer 0 node:

```http
GET /v1/{context_id}/entities?resolves_to=knowledge:product:airpods_pro_2
```

To find unlinked entities that might benefit from resolution:

```http
GET /v1/{context_id}/entities?resolves_to=null&type=Product
```
