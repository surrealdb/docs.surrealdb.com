# Coherence, retrieval, and cost tiers

## Five coherence dimensions

Memory is coherent along five axes at once – Spectron stores enough metadata to answer questions on each:

| Dimension | What it gives you |
| --- | --- |
| **Semantic** | Similarity before structure is explicit: embedding-based recall over the unified substrate (`entity.embedding`, passage embeddings). |
| **Lexical** | What was actually said or shown, down to byte offsets: extracted attributes carry `source.span: { start, end }` into the originating turn or document passage. Citations are a **stored field**, not best-effort prose. |
| **Relational** | Understanding as connections: a single `entity` / `relation` graph so entailment, contradiction, and corroboration are **composable** graph operations. |
| **Time** | What held when, and how beliefs evolved: `valid_from` / `valid_until` supersession chains; `as_of` and `VERSION`-style queries for belief history versus byte-level substrate history. See [Tri-temporal model](/docs/spectron/architecture/tri-temporal-model). |
| **Space** | Where a fact was captured, where it applies, and how facts relate geographically – optional geometry on entities, sessions, chunks, and sources; geo filters compose with semantic and graph signals in the **same** fused ranker. |

Vector-only approaches tend to fail on several of these at once; row-only stores fail on all five if you never add structure. Spectron is designed to cover the set **by storage**, not by rediscovering structure on every query.

## Structural retrieval (beyond embeddings)

Retrieval is **hybrid by design**. Embeddings are one signal; they are fused with other precomputed structure so top‑k is not a black box.

Typical signals in the fused ranker include:

- **Vector recall** – dense embeddings on entities, attributes, chunks, and (when enabled) modality-native vectors for images and audio.
- **Lexical recall** – BM25 over chunk text and entity surface forms for exact phrases and rare terms.
- **Graph traversal** – limited hops from seed entities to surface related facts when surface forms diverge.
- **Keyword bridges** – RAKE-derived `keyword` nodes with scored edges (including PMI) so rare discriminative terms still connect distant chunks.
- **Section embeddings and document links** – rollups and chunk–chunk / document–document edges so retrieval can return **related sections**, not only the single nearest chunk.
- **Personalised PageRank** – graph-walk scoring biased toward query seeds, as a structural feature in the ranker.
- **Geographic recall** – radius, polygon, nearest‑k predicates on stored geometry, composed with the other signals.
- **Trace-derived features** – prior `retrieval_trace` outcomes influence future ranking (boost what worked; demote what led to corrections).

Each `/query` emits a **`retrieval_trace`** recording candidates, per-signal contributions, and the returned set – so any hit is auditable as a **weighted combination** of signals, not an opaque embedding top‑k.

For hands-on retrieval modes and APIs, see [Hybrid search](/docs/spectron/knowledge/retrieval/hybrid-search) and related pages under [Knowledge](/docs/spectron/knowledge).

## Tiered query resolution

Spectron does **not** run the same expensive path on every request. Reads route through a **four-tier** ladder after a query-understanding step:

| Tier | What happens | Cost profile |
| --- | --- | --- |
| **1 – Direct structured lookup** | Typed questions resolved straight from the `entity` / `attribute` graph by key – no embeddings, no LLM, no ranking pass. | Sub‑millisecond when it applies. |
| **2 – Response reuse** | Semantic match against prior **`response_trace`** rows in the same Context and scope, with **entity-aware invalidation** (cited facts must still be current). Returns a prior answer and links a fresh trace via `reused_from`. | Tens of milliseconds; no generation on a hit. |
| **3 – Hybrid retrieval and synthesis** | The “normal” path: fused retrieval then LLM synthesis. | Hundreds of milliseconds – the default for open questions. |
| **4 – Full-context fallback** | Broader sweep when tier 3 is thin or low-confidence: more candidates, deeper graph hops, optional query rewrite, longer context – **explicit** escalation. | Higher token cost and latency; still traceable as tier‑4. |

Tiers cascade (miss on 2 falls to 3; thin 3 escalates to 4). Each tier writes **`retrieval_trace`** metadata describing which tier ran and why – so tier mix is observable and tunable per Context.

This is the **cost story made literal**: cheap questions stay cheap; expensive work is visible in traces instead of disappearing into a flat per-request average.
