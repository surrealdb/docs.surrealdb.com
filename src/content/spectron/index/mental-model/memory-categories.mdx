# Memory categories

Layer 1 memory is divided into five categories. Each category represents a different kind of information with different volatility characteristics, decay rates, and retrieval behaviour. Assigning records to the correct category is how Spectron keeps recent situational context from crowding out stable biographical facts.

## Overview

| Category | Volatility | Decay rate | Storage | Retrieval boost |
|---|---|---|---|---|
| Identity | Low | None | `attribute` table | 1.0 |
| Knowledge | Medium | 0.995 / day | `attribute` table | 0.8 |
| Context | High | 0.95 / day | `attribute` table | 0.5 |
| Instructions | Varies | None | `instruction` table | — |
| Unknowns | Varies | None | `uncertainty` table | — |

## Identity

**Identity** captures who someone is: stable biographical and organisational facts that rarely change.

Examples:
- Name, preferred name, and pronouns
- Role and seniority
- Company or organisation
- Location or timezone
- Communication style preferences

Identity attributes carry **no decay** — they remain at full relevance indefinitely unless explicitly corrected. The retrieval boost of **1.0** means identity facts are always surfaced at full weight when they match a query.

```json
{
  "entity": "entity:[\"Person\", \"tobie_morgan_hitchcock\"]",
  "category": "identity",
  "key": "role",
  "value": "Co-founder",
  "valid_from": "2026-01-15T09:00:00Z",
  "source_turn": "turn:xyz789"
}
```

## Knowledge

**Knowledge** captures learned preferences, opinions, expertise, and domain understanding that evolve over time but do not change rapidly.

Examples:
- Technology preferences ("prefers TypeScript over JavaScript")
- Expertise areas ("experienced with distributed systems")
- Communication preferences ("prefers detailed explanations")
- Opinions on products, tools, or approaches

Knowledge attributes decay at **0.995 per day** — a very slow rate that models the gradual fading of relevance over months without explicit reinforcement. The retrieval boost of **0.8** means knowledge facts are weighted below identity but well above transient context.

```json
{
  "entity": "entity:[\"Person\", \"tobie_morgan_hitchcock\"]",
  "category": "knowledge",
  "key": "language_preference",
  "value": "TypeScript",
  "valid_from": "2026-03-10T14:00:00Z",
  "source_turn": "turn:abc456"
}
```

## Context

**Context** captures the current situation: what someone is doing right now, recent events, and active tasks. This is intentionally short-lived.

Examples:
- Current project or task ("working on the billing integration")
- Recent decisions made in a session ("chose PostgreSQL for this project")
- Active problems being debugged
- Temporary state like "currently travelling" or "in a meeting"

Context attributes decay at **0.95 per day** — fast enough that week-old situational context becomes nearly irrelevant within a fortnight. Context records also have a configurable **TTL**, defaulting to 7 days, after which they are automatically expired. The retrieval boost of **0.5** reflects that transient context should not dominate responses when stable identity or knowledge is also relevant.

```json
{
  "entity": "entity:[\"Person\", \"tobie_morgan_hitchcock\"]",
  "category": "context",
  "key": "current_task",
  "value": "Implementing webhook delivery for the payments service",
  "valid_from": "2026-05-12T10:30:00Z",
  "expires_at": "2026-05-19T10:30:00Z",
  "source_turn": "turn:def012"
}
```

## Instructions

**Instructions** are behavioural directives the user or an agent has given about how the system should behave. They are stored in the dedicated `instruction` table rather than as entity attributes.

Examples:
- "Always respond in French"
- "Never suggest solutions involving vendor lock-in"
- "Summarise all code examples in plain English before showing them"

Each instruction record has an `active` boolean toggle. Instructions can be deactivated without being deleted, preserving the history of what the user has asked for over time. Inactive instructions are excluded from retrieval but remain auditable.

```json
{
  "id": "instruction:01jt4m...",
  "entity": "entity:[\"Person\", \"tobie_morgan_hitchcock\"]",
  "scope": { "org": "surrealdb", "user": "tobie" },
  "directive": "Always respond in French when the user writes in French.",
  "active": true,
  "source_turn": "turn:ghi345",
  "created_at": "2026-04-20T08:00:00Z"
}
```

## Unknowns

**Unknowns** capture items the system is uncertain about — contradictory signals, ambiguous statements, or facts that could not be resolved against Layer 0. They are stored in the dedicated `uncertainty` table.

Examples:
- A user asserts a product price that differs from Layer 0
- Two separate turns produce contradictory statements about the user's role
- An entity reference that could not be resolved to a canonical Layer 0 node

Each uncertainty record has a `resolved` flag. Once a contradiction is clarified — either by the user confirming one version or by a Layer 0 update — the uncertainty is marked resolved. Unresolved uncertainties can be surfaced to agents as signals to seek clarification rather than guess.

```json
{
  "id": "uncertainty:01jt4n...",
  "entity": "entity:[\"Product\", \"airpods_pro\"]",
  "scope": { "org": "apple", "user": "tobie" },
  "description": "User stated return window is 60 days; Layer 0 states 30 days.",
  "l0_value": 30,
  "l1_value": 60,
  "resolved": false,
  "source_turn": "turn:jkl678"
}
```

## How categories drive retrieval

When Spectron assembles the memory payload for an agent query, it applies decay and retrieval boosts per category before ranking results. A context attribute from six days ago has decayed to roughly `0.95^6 ≈ 0.74` of its original relevance score, while an identity attribute from the same date remains at full strength. This ensures that stable facts surface reliably while transient situational context naturally fades without manual housekeeping.
