# Two-layer architecture

Spectron organises knowledge into two distinct layers with different authorities, different volatility characteristics, and different ownership semantics. Understanding this separation is the foundation for reasoning about how Spectron behaves.

## Layer 0 — Knowledge

Layer 0 holds **authoritative organisational knowledge**: the ground truth your business or product team controls. This includes:

- Product catalogues, pricing, and specifications
- Policies (return policies, terms of service, support SLAs)
- FAQs and troubleshooting guides
- Internal documents and process definitions
- Any structured data you want agents to treat as canonical

Layer 0 content is typically loaded through the ingestion API or a data pipeline. It does not change based on what users say — it changes only when you explicitly update it.

## Layer 1 — Memory

Layer 1 holds **conversational memory**: facts learned from interactions between agents and users. This includes:

- Who the user is and what they have told the system about themselves
- Preferences, opinions, and past decisions
- Current situation and ongoing tasks
- Behavioural instructions the user has given
- Uncertainties and contradictions the system has detected

Layer 1 evolves continuously. Every turn in a conversation can produce new entities, attributes, and relations.

## The authority relationship

**Layer 0 always wins.** When a user asserts something that contradicts a Layer 0 fact, Spectron records the user's belief in Layer 1 but flags it as contradicting the authoritative record — it does not overwrite Layer 0.

For example: a user says "your return policy is 60 days". If Layer 0 states 30 days, Spectron stores the user's belief as a Layer 1 attribute on the relevant entity with a `contradicts_l0` flag, and the agent is informed of the discrepancy. The Layer 0 fact remains untouched.

This prevents hallucination amplification: an agent that keeps hearing incorrect facts from users will never promote those facts into the authoritative record.

| | Layer 0 | Layer 1 |
|---|---|---|
| Content | Documents, policies, product data | Conversational facts, preferences, instructions |
| Authority | Canonical — cannot be silently overridden | Contextual — lower authority, evolves over time |
| Ownership | Your team / ingestion pipeline | Agents and user interactions |
| Mutability | Explicit updates only | Updated every turn |
| Decay | None | Category-dependent decay rates |

## How Layer 1 links back to Layer 0

Layer 1 entities can carry a `resolves_to` relation that points to the corresponding Layer 0 knowledge node. This is how a conversational reference to a product name connects to authoritative product data.

Consider the following scenario: a user says "I bought AirPods Pro". Spectron creates a Layer 1 entity for the product the user mentioned, then resolves it to the canonical Layer 0 node.

```json
// Layer 1 entity — created from the conversation
{
  "id": "entity:[\"Product\", \"airpods_pro\"]",
  "layer": 1,
  "scope": { "org": "apple", "user": "tobie" },
  "source_turn": "turn:abc123"
}

// Layer 0 knowledge node — loaded from your product catalogue
{
  "id": "knowledge:[\"Product\", \"airpods_pro\"]",
  "layer": 0,
  "price": 279,
  "return_window_days": 30,
  "warranty_years": 1
}

// resolves_to relation — connects the two layers
{
  "in": "entity:[\"Product\", \"airpods_pro\"]",
  "out": "knowledge:[\"Product\", \"airpods_pro\"]",
  "relation": "resolves_to"
}
```

When an agent queries for information about AirPods Pro, Spectron traverses the `resolves_to` relation and surfaces both the Layer 1 context (the user owns one) and the Layer 0 facts (price, return policy, warranty). The agent receives a complete picture without needing to treat the layers as separate concerns at query time.

## Why the separation matters

**Accuracy**: Authoritative facts cannot be corrupted by conversational noise. An agent serving thousands of users will encounter incorrect assertions constantly; the layered model prevents any single interaction from poisoning the knowledge base.

**Observability**: Because corrections are recorded rather than applied, you can inspect exactly what users believe versus what is true, and audit every discrepancy.

**Flexibility**: Layer 1 can be scoped tightly to individual users or organisations while Layer 0 remains globally accessible within a Context. See [Contexts and scope](/spectron/mental-model/contexts-and-scope) for how scoping works across both layers.
