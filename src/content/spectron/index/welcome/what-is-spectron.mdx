# What is Spectron?

Spectron is an **accurate, observable, conversational knowledge system** for AI agents. It provides a unified memory layer where agents store, retrieve, and evolve knowledge across sessions — backed by SurrealDB's multi-model engine.

## The problem

Most AI agents are stateless. Each session starts from scratch: the agent has no memory of previous conversations, no knowledge of who the user is, no awareness of decisions that were made last week. This is not a limitation of the language models themselves — it is a missing infrastructure layer.

The naive fix is to embed conversation history into context windows. This breaks down quickly:

- Context windows have limits; old turns get truncated.
- Stuffing raw history into a prompt is expensive and slow.
- There is no deduplication — the same fact appears dozens of times.
- Contradictions accumulate silently. There is no correction mechanism.
- Nothing is retrievable by query; the agent must read everything to find anything.

A more common fix is to use a vector store: embed past turns, retrieve the most similar chunks at query time. This improves recall but introduces a different set of problems, which are covered in [Why agentic memory?](/spectron/welcome/why-agentic-memory).

## What Spectron does differently

Spectron treats memory as a **structured knowledge graph**, not a bag of text chunks. When a turn arrives, Spectron extracts discrete facts — entities, attributes, and relations — and stores them with full provenance. Every fact knows:

- Which session and turn it came from (`source_turn`)
- When it was valid (`valid_from` / `valid_until`)
- Whether it supersedes a previous fact (supersession chain)
- What dimensional scope it applies to (organisation, user, project)

This makes memory **observable**. You can query the memory state directly, see exactly what the agent knows about a user, inspect when a fact changed and why, and verify that the system is correct — without trusting that the retrieval layer is returning the right things.

## The design question

Spectron's design is guided by a single question:

> Can someone watch the memory state in real-time and verify that the system is correct?

If the answer is no — if memory is an opaque blob of embeddings with no auditability — then it is impossible to know whether the agent is operating on accurate information. Spectron is built to make that question answerable.

## The two-layer model

Spectron organises knowledge into two layers that serve different purposes and have different authority:

### Layer 0 — Knowledge

Layer 0 holds authoritative organisational knowledge: documentation, policies, product data, FAQs, and any structured data ingested from external systems. It is populated through file uploads or connectors (Snowflake, Databricks, Google Drive, Notion, GitHub, Confluence) and supports multiple content formats — text, Markdown, JSON, HTML, PDF, images, audio, and video.

Layer 0 is **read-only from a conversational perspective**. It is updated deliberately by operators, not extracted from user messages.

### Layer 1 — Memory

Layer 1 holds everything the agent learns from conversations: facts stated by users, inferred attributes, relationships between entities, behavioural instructions, and uncertain or contradictory items flagged for review.

Layer 1 is **written automatically** by the extraction pipeline on every turn.

### Authority hierarchy

When Layer 1 conflicts with Layer 0, **Layer 0 wins**. A user claiming that a product costs something different from the official price sheet does not update the authoritative record. The conflict is logged, but the knowledge layer is protected from conversational drift.

## Contexts

A **Context** is the unit of isolation. Each Context bundles its own Layer 0 and Layer 1 inside a single SurrealDB `(namespace, database)` pair. Different products, tenants, or teams operate in separate Contexts with no data leakage between them.

Every request is routed through `/v1/{context_id}/...`, so the context boundary is enforced at the API level.

## What you can build

Spectron is a general-purpose memory layer. Typical uses include:

- Personal AI assistants that remember user preferences across sessions
- Customer-facing agents that accumulate product knowledge and interaction history
- Coding agents that track project-specific conventions and past decisions
- Multi-agent systems where agents share a common memory pool
- Any application where an LLM needs accurate, queryable, long-term context

The [Cookbooks](/spectron/cookbooks) section has worked examples for each of these patterns.
