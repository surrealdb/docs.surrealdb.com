# Transactions

The v2 SDK exposes server-side transactions through the connection controller. Transactions are a WebSocket feature and require SurrealDB `3.0.0` or later.

> [!WARNING]
> The v2 SDK is an early **alpha**. Transaction handling is advanced API and may change before a stable release.

> [!IMPORTANT]
> Server-side transactions require a WebSocket connection and SurrealDB `3.0.0` or later. The SDK gates this feature at runtime and throws an `UnavailableFeatureException` on older servers, or an `UnsupportedFeatureException` on the HTTP engine.

## Begin, commit, and cancel

Transaction control lives on the `ConnectionController`, accessed via `connection()`. `begin()` returns a transaction id; pass it to `commit()` or `cancel()`.

```php
$connection = $db->connection();

$txn = $connection->begin();

try {
    // Run statements as part of the transaction.
    $connection->query(
        new \SurrealDB\SDK\Query\BoundQuery('CREATE person SET name = $name', ['name' => 'Tobie']),
        session: null,
        txn: $txn,
    );

    $connection->commit($txn);
} catch (\Throwable $error) {
    $connection->cancel($txn);
    throw $error;
}
```

## Inline transactions with SurrealQL

You can also run a transaction as a single SurrealQL batch with `run()`. This works over both engines and on any supported server version, because the transaction is expressed entirely in SurrealQL:

```php
$db->run(<<<'SURQL'
BEGIN TRANSACTION;
CREATE account SET balance = 100;
UPDATE account SET balance -= 10;
COMMIT TRANSACTION;
SURQL);
```

> [!NOTE]
> The [Surqlize ORM](/docs/languages/php/orm/surqlize/transactions) provides a higher-level transaction helper that batches ORM queries into a single `BEGIN/COMMIT` block. That is a different mechanism from the SDK's server-side transaction ids described here.
