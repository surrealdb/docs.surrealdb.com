# Connection lifecycle

The `SurrealDBManager` and `SurrealFactory` manage the connection lifecycle based on your configuration.

> [!WARNING]
> This package depends on the **alpha** v2 SDK. Its behavior may change.

## Lifecycle stages

1. **Lazy client creation.** `client($name)` builds a `Surreal` instance per connection name and caches it for the lifetime of the container.
2. **Connect on resolve.** When `lifecycle.connect_on_resolve` (or `auto_connect`) is true, the factory connects immediately when the client singleton is first built.
3. **Lazy connect on query.** The manager's `run()` and `query()` connect first if the client is not already connected.
4. **Health check.** When `health_check_on_resolve` is true, the manager runs a health check after connecting.
5. **Disconnect on terminate.** When `disconnect_on_terminate` is true, the SDK provider registers an `app->terminating()` callback that closes connections at the end of the request.

## Lifecycle configuration

```php
'lifecycle' => [
    'connect_on_resolve' => env('SURREALDB_CONNECT_ON_RESOLVE', env('SURREALDB_AUTO_CONNECT', true)),
    'disconnect_on_terminate' => env('SURREALDB_DISCONNECT_ON_TERMINATE', true),
    'health_check_on_resolve' => env('SURREALDB_HEALTH_CHECK_ON_RESOLVE', false),
],
```

## Deferred connection

To render pages without connecting eagerly, set `SURREALDB_AUTO_CONNECT=false` and connect on demand:

```php
use SurrealDB\Laravel\SurrealDBManager;

app(SurrealDBManager::class)->connect();
```

## Long-running processes

> [!NOTE]
> The manager is a container singleton. Under PHP-FPM that is normally once per request, but under Octane, queue workers, or long-running commands the client lives longer. Tune `disconnect_on_terminate` and the lifecycle settings accordingly, and reconnect when a worker's connection has been closed.
