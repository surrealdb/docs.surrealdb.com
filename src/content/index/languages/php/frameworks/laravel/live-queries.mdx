# Live queries

The Laravel package does not wrap live queries. Use the raw [SDK client](/docs/languages/php/v2) over a WebSocket connection.

> [!WARNING]
> This package depends on the **alpha** v2 SDK. Its API may change.

> [!IMPORTANT]
> Live queries require a WebSocket connection (`ws://` or `wss://`) and depend on the SurrealDB server version. They are not available over HTTP.

## Consuming a live query

Start a live query with `LIVE SELECT`, then iterate notifications by id.

```php
use SurrealDB\SDK\Surreal;

$surreal = app(Surreal::class);

$uuid = $surreal->run('LIVE SELECT * FROM pulse_event')[0];

foreach ($surreal->live($uuid) as $message) {
    // $message->action  (Create | Update | Delete)
    // $message->record
    // $message->value
}
```

## Streaming and worker concurrency

The `live()` iterator blocks until the next notification arrives, so it holds one worker for the lifetime of the subscription. This matters for HTTP applications and server-sent events (SSE).

> [!WARNING]
> You must run the PHP runtime with multiple workers when serving live queries. A blocking `live()` read occupies a worker for as long as the stream is open, so with a single worker the server freezes and the entire application becomes unusable until the stream ends. Use a multi-worker `php artisan serve` (for example `PHP_CLI_SERVER_WORKERS=4 php artisan serve`), [Laravel Octane](https://laravel.com/docs/octane), or FrankenPHP. PHP-FPM should be configured with enough workers to spare at least one per concurrent stream.

> [!NOTE]
> For a long-lived stream, build a dedicated short-lived client with `SurrealFactory::create()` rather than reusing the shared request-scoped client; a blocking `live()` read can otherwise leave the shared connection mid-frame.

Operational guidance:

- Use a WebSocket connection, and a SurrealDB server version that supports live queries.
- Run multiple PHP workers so a blocked stream never starves the rest of the application. A single-worker `php artisan serve` also cannot publish changes while a stream is open.
- Consider a polling fallback for robustness across environments.
