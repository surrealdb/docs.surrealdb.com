# Distributed live queries

**Live queries** let clients subscribe to result sets that update as underlying data changes. In **Community Edition**, live query behaviour is oriented around a **single** SurrealDB node: subscriptions are served by the instance your client is connected to.

**SurrealDB Enterprise Edition** adds **distributed live queries**: the same subscription semantics can be coordinated across **multiple nodes** in a cluster so that relevant changes originating on different members still propagate to subscribers, subject to your topology and consistency settings.

That distinction matters for horizontally scaled workloads: application tiers can remain simple—using the same `LIVE` patterns—while the cluster shares responsibility for routing change notifications.

Operational teams should still design for **network partitions** and **failover**: validate client reconnect behaviour, monitor subscription churn, and align retention with your SLAs. Pair this feature with EE security tooling such as [Audit logging](/docs/manage/enterprise/security/audit-logging) when you need an evidence trail for access and changes.
