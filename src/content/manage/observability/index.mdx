# Observability overview

A SurrealDB deployment can be observed in several complementary ways:

- [Logging](/docs/manage/observability/logging) — server log level, text or JSON format, files, sockets, and line-based slow-query logging (no metrics stack required).
- [Observability (metrics and Prometheus)](/docs/manage/observability/observability) — unified OpenTelemetry metrics, `GET /metrics` (pull), naming, migration from pre-3.1 series, the public metric allowlist, and guidance that applies to **Community and Enterprise**.
- [Telemetry (OTLP)](/docs/manage/observability/telemetry) — push export when `SURREAL_TELEMETRY_PROVIDER=otlp` is set: metrics, logs, traces, intervals, and backward-compatible HTTP/RPC instruments.
- [Enterprise observability](/docs/manage/observability/enterprise-observability) — what **SurrealDB Enterprise** adds on top: SurrealDS cluster metrics, audit and slow-query file pipelines, pipeline self-metrics, OTLP log export opt-ins, and operator tiering.
- [Tokio console](/docs/manage/observability/tokio-console) — optional Tokio runtime debugging (tasks, poll times) for deep performance work alongside metrics and traces.

**Pull** (`GET /metrics`) means scrapers call your server. **Push** (OTLP) means the server opens an export connection to a collector. Both paths share the same instruments from SurrealDB 3.1 onward.

Canonical reference prose is maintained next to the engine: open-source trees ship `doc/OBSERVABILITY.md` and `doc/TELEMETRY.md`; the SurrealDB Enterprise distribution ships an extended `doc/OBSERVABILITY.md` with **[C]** / **[E]** catalogue markers. This documentation section mirrors those three angles for the website.
